# 样板:一棵填好的记忆树 + 怎么新增一条记忆

> 这是一个**编造的示例项目**(WebNote,一个笔记 App),给你**照着抄结构**——因为空有抽象规则、没有填好的实物,最难上手的就是"叶怎么建、怎么挂、怎么去重、归哪类"。

## 这棵树长这样

```
🗺️ 结构 webnote-structure.md         ← 项目真实代码「形状」(目录/模块/入口)= 锚点底图
🌲 根   webnote-root.md              ← 项目身份 + 铁律 + 模块(枝)索引
 └─🌿 枝  webnote-sync.md             ← 一个模块一张「逻辑地图」(核心逻辑 + 触手)
     └─🍃 叶  webnote-sync-conflict-dedup.md   ← 一个具体事件(问题/根因/结论),链回它的地图
```

- **结构树** = 项目真实代码的**形状**(目录/模块/入口/继承链),客观、可核对。它是「锚点优先」的底图:新记忆归位前先来这查"它讲的东西在代码哪儿"。第一次整理时扫代码建它、之后增量更新。
- **根** = 一个仓库一个。列出所有模块(枝)。
- **枝(地图)** = 一个代码模块一张。写这个模块**怎么运作(核心逻辑)+ 它连着谁(触手)**。
- **叶(事件)** = 一件具体的事(bug/决策/判断)。**每片叶必须声明 `模块地图 = [[它的枝]]`**,否则就是孤儿。

跑 `../tools/check-tree.sh` 能验:每片叶都链回了地图、每张地图都在根的索引里、没孤儿。

## 树结构的底层逻辑:锚点优先(anchor-first)

归类/去重/挂载难,是因为容易**凭感觉发明分类**。更稳的办法 = **每条记忆的地址是"推导"出来的,不是"发明"的**:

- **归哪儿 = 它讲的「代码锚点」在哪儿** —— 根=仓库,枝=代码模块(`src/sync/`),叶=某段代码的具体事件。**在真实代码里找到它讲的东西,那就是它的家**。→ 把"主观分类"换成"客观定位"。
- **重不重 = 锚点 + 断言** —— 讲同一个东西 + 说同一件事 = 重复 → **合并**,别新建。
- **是枝还是叶 = 锚点的作用域** —— 整个模块的核心逻辑 → 枝(地图);某个具体问题/决策 → 叶(事件)。

**一句话:先问"它讲代码里哪个东西",别先问"它属于哪一类"。**

> (锚不到代码的知识——比如跨模块决策、用户偏好——就用一个"领域概念"当锚点,同样是先定位再归位。)

## 新增一条记忆:决策流程(照着走,别凭感觉)

学到一条值得记的东西时:

1. **找锚点** = 它在讲代码里哪个东西?(某文件/模块/系统/整个项目)→ 在真实代码里定位它。
2. **查重(锚点 + 断言)** = 搜现有记忆里**同锚点**的:
   - 同锚点 + 同一件事 → **更新/合并旧的**,别建新;
   - 同锚点 + 不同的事 → 建**兄弟**;
   - 全新锚点 → 建新的。
3. **定层级** = 看锚点作用域:整模块核心逻辑 → **地图(枝)**;具体一件事 → **叶(事件)**。
4. **挂载**:叶声明 `模块地图 = [[它的地图]]`;新枝登记进根的模块索引;关键决策的原始对话 → 血脉日志编号 + 叶里 `🩸对话▸N` 索引。
5. **过一遍不矛盾**:和现有记忆有没有逻辑打架?有 → 按时间新者改对(见 `../no-contradiction-when-writing-memory.md`)。
6. **验**:跑 `../tools/check-tree.sh`。
