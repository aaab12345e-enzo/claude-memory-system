**每日记忆整理("日清")规则(用户 2026-06-17 定,核心规则):**

目标:用户要**每天一个全新工作台**且知识不丢。每天做一次日清(意图是 0 点;实际在当日收尾或次日开场的会话边界触发——见末尾"自动化现实")。

**★核心目的(记牢):保证记忆库整体逻辑自洽、无明显矛盾——与权重高低无关。** 下面的合并/加权/查代码都只是**手段**;具体怎么做我自己拿捏,用户可在往后日子里帮我慢慢优化。

**步骤:**
1. **落当前窗口** —— 当前窗口里还没入库的重要结论/决策/未完线头,全部存进记忆(走 [[checkpoint-key-memory-after-each-task]])。
2. **合并去重 + 加权** —— 通查所有记忆,凡**逻辑类似 / 同类错误 / 同主题版本更迭(新取代旧)** 的 → **合并成一条**、删被取代的旧条;合并即"该主题反复出现",按 [[memory-weighting-rule]] **累加权重**。 **★含元记忆(用户 2026-06-25·洞2)**:习惯钩子链 / 成长 / 结构规则这些"记忆的记忆"也**一并按结构修剪**(合并重叠钩子、删过时规则、一事实一处)——元记忆同样会肿,别只清内容记忆。
3. **矛盾校真(不限权重)** —— 记忆库里**任何**记忆之间、或记忆与现状有逻辑矛盾,都要消除,**不分权重高低**。仲裁:**翻代码(repo)看现状 + 时间较新者为准,把记忆改到最新正确**(高权重≠对)。**并列动作:通读第二层对话记录去矛盾**——用户前后发言冲突(改主意/纠正/版本更迭)的,以时间较新者为准、旧条旁标 `→已被X取代`。
4. **重排索引** —— MEMORY.md:核心规则 → ⚑高权重(先读) → 常规,`⚑N` 权重同步。
5. **推备份** —— 整理完,把记忆库 push 上云,并提醒/顺手推项目库;否则换机丢最新记忆。**注:push memory 库这步若我(agent)跑会被安全分类器拦"记忆外传",让用户在终端手动跑。**
6. **清台开新** —— 整理完,**开一个全新工作台**(新会话);旧窗口记录可关/清,因为重要信息已全入库、无损接续。
7. **结尾提记忆改进方案(用户 2026-06-23 加)** —— 每次日清**结尾**,若我对记忆系统本身(结构/规则/分层/权重/检索/去重方式…)有更好的改进想法,主动提出来给用户过(同 [[confirm-plan-before-acting]] 的「主动提更优解」精神,只是对象是记忆系统)。没有就说没有,别硬凑。

## 如何判断 类似/重复/版本更迭/矛盾(库整理判据)
每条记忆抽成 **主题(讲哪个系统/文件/概念/规则)+ 断言(说什么是真 / 该怎么做)+ 范围 + 时间/权重**;比的就是「主题 + 断言」。判**主题相同**的抓手:description 指同一对象、正文引同一 file:line / 字段 / id / 概念、`[[links]]` 互指或同组、关键实体重叠。
- **类似/重复** = 主题同 + 断言一致或互相包含 → **合并**(并集细节、留最强表述、**权重相加**、时间取最新、删旧条)。
- **版本更迭** = 主题同,断言不同是因**现实变了**(代码改/决策变/数据更新),旧的曾对、被新的取代 → **替换**(新为准,并入旧条里仍成立的细节,删旧)。**这不是矛盾,别去"纠正",直接迭代。**
- **矛盾** = 主题同,两断言**逻辑不能同真**且都自称"当下成立"(非一旧一新) → **必须消灭**:① 翻代码/repo 看现状;② 代码定不了看时间(新者为准);③ 改对错的那条、保留权重。**高权重≠对**。
- **迭代 vs 矛盾 的判别**:能用「旧→新替代」时间线解释 → 迭代;两者都想当下为真且互斥、调和不了 → 矛盾(查代码定真伪)。

## 第二层记忆:四类
**① 踩坑档案 `pitfalls/`** 记坑避坑(下文,按需拉);**② 对话记录** 逐条留底 用户的话+前后文(精选,按需拉,日清去矛盾见步骤3);**③ 遗留问题日志** 只记 OPEN 项(用户 2026-06-20 加);**④ 全程对话记录 `conversations/<日期>.md`** —— ⚠️**已被 ⑤ 的 checkpoint 直写取代、冗余可退役**(用户 2026-06-27):原设计是先倒一份全程再事后分拣,现改成直接写进各主题日志,不再需要这个中转大文件。**⑤ 事件血脉 `conversations/events/<事件slug>.md` = 纯时序编号日志(用户 2026-06-27;模版见 [[memory-topic-by-project-structure]] 事件模版·日志+索引)** —— **checkpoint 直写**:每干完一块/定稿一决策**当场**写——血脉追一条编号 `[N]`(🗣️用户原话全留+💬我精简)、叶对应 `## 功能点` 更新 `🩸对话▸N` 索引。**不分钉住/过程区**(分类交给叶索引,日志只管纯记)。`conversations/<日期>.md` 全程大文件**冗余、可退役**。日清只做:**补漏没直写的 + 去重校真**(不当搬运工)。**跨会话记忆主题** → 专门日志 `conversations/events/记忆模型优化.md`(判据:改我记忆/习惯/行为=记这、改代码/数据=工作事件)。**加载=按叶索引精准取那几条**(决策再老也丢不了)。→ 治"提炼丢掉怎么聊出来的/丢了昨天对话" + "读最新N条会丢早期定稿"。——**新窗口开场先读**(故 MEMORY.md 顶部留钩子,非纯按需),**日清更新**:加本窗口新 open、删已解决/废弃/迭代的、与一层"待/剩"对账去重(本日志=open 项唯一权威);**更新定稿前把清单跟用户过一遍确认"是否属实"**(还成立/已解决/已变)再落定。

## 踩坑档案 pitfalls/(重点:记坑避坑,不是回放步骤)
**作用**:碰到**类似主题**的新需求时,据此避开以前踩过的坑。每个实质任务一份 `memory/pitfalls/<日期>-<任务slug>.md`,**不进 MEMORY.md**(按需拉);从相关主题记忆正文挂一行 `▸踩坑: pitfalls/xxx.md`,靠**主题召回 → 顺链翻坑**。
**每个坑的结构(★第4行最值钱):**
> 我以为/做了 · 怎么暴露的 · 真因 · **★为啥会上当(误导信号——下次靠它提前识别)** · 改法 · (升华成精炼记忆?[[..]];复发则权重+1)
**约定**:① 标日期;② 坑的教训比命令耐放,但仍"复用前核现行代码";③ **日清只读取/挂链 pitfalls,不改写它**(它是历史现场,改了就失真——和精炼记忆"查代码纠矛盾"区别对待);④ **复发或重要的坑 → 升华成一条精炼(常高权重)记忆当"避坑铁律",权重+1**(踩坑档案喂养权重系统)。

**自动化现实(诚实,别假装全自动):**
- 「整理当前窗口内容」必须由**当前这个会话**做——定时器/别的会话读不到本窗口的实时对话。
- 「清空/重置工作台」**没有定时器能替你清掉正在用的窗口**;"全新工作台" = 你开个新会话(日清已让它无损)。
- 能纯定时跑的只有「记忆库整理(合并/加权/查代码纠错)」这块,且要**本地**调度(记忆在本地,云端 routine 碰不到本地记忆)。
- 实操:每天在会话边界喊一句"**日清**"即按本规则执行(= 这次"重启前落记忆"的加强版)。
- **无人值守 headless 自动改记忆被拦,但会话内定时排程通了(2026-06-29 实测更新)**:① ❌ 无人值守、单独 headless 会话、关审批门 → 被安全分类器拦("未授权持久化+关审批门的自主agent循环")。② ✅ **会话内定时任务(durable)在凌晨真 fire 进了当前活会话、分类器没拦、正常做了日清** —— 关键区别:**它在当前会话里把 prompt 排进来(不是另起 headless、审批门照常),又能碰本地记忆**。结论修正:**真·定时日清这条路 = 会话内排程通**(不是无人值守 headless)。详见 [[memory-system-architecture]] §三定时触发器 + [[claude-classifier-blocks-agent-privileged-actions]]。
