> Example · feedback-type memory: this is a sample of "recording your user's preferences" — swap it out for your own user's actual preferences.

The user's team communicates day-to-day via chat tools (short messages, conversational). Anything for team members should be made into a **short, light, colloquial** form that can be pasted straight into chat — no tables / emoji severity tags / `file:line` scattered everywhere / multi-section long markdown docs — none of which any audience is comfortable reading (the non-technical side finds it too technical, the technical side finds it too verbose).

**Why:** the communication medium is chat, not a doc repository; the audience may split into different roles (some want business language, some want distilled technical points), and a long formal doc matches neither.

**How to apply:** by default, give a few sentences that "can be forwarded straight to the group chat" — don't spin up a big doc on your own initiative. When you need to split by audience, give a short one to each: business language for the business side, key changes + where for the technical side. When something needs to be archived, then write up a doc, but keep it as lean as possible too. Related: [[vscode-chat-links-dont-open]] (when referencing files, give the path, don't make them click a link).
