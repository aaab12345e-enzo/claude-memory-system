**The daily memory-consolidation ("day-clean") rule (set by the user 2026-06-17, a core rule):**

Goal: the user wants **a brand-new workspace every day** and no knowledge lost. Do one consolidation per day (intended for midnight; in practice triggered at the session boundary when wrapping up the day or opening the next day's session — see "the reality of automation" at the end).

**★ Core purpose (memorize it): keep the memory library internally consistent and free of obvious contradictions overall — regardless of weight.** The merging/weighting/code-checking below are all just **means**; how exactly I do it is my own call, and the user can help me refine it gradually over the coming days.

**Steps:**
1. **Flush the current window** —— all the important conclusions/decisions/loose threads in the current window that aren't yet in the library get stored into memory (via [[checkpoint-key-memory-after-each-task]]).
2. **Merge, dedup + weight** —— go through all memories; wherever they are **logically similar / the same kind of mistake / the same topic superseded across versions (new replaces old)** → **merge into one entry**, delete the superseded old entry; merging means "this topic recurs," so **accumulate weight** per [[memory-weighting-rule]]. **★ Includes meta-memory (user 2026-06-25 · insight 2)**: "memory about memory" like the habit-hook chain / growth / structural rules also gets **pruned structurally along with the rest** (merge overlapping hooks, delete obsolete rules, one fact in one place) — meta-memory bloats too, don't only clean content memory.
3. **Contradiction verification (regardless of weight)** —— any logical contradiction between **any** memories in the library, or between a memory and the current reality, must be eliminated, **irrespective of weight**. Arbitration: **check the code (repo) for current reality + the more recent one wins, and fix the memory to be the latest correct** (high weight ≠ correct). **In parallel: read through the second-layer dialogue records to remove contradictions** — where the user's statements conflict over time (changed their mind / corrected / version turnover), the more recent one wins, and mark the old entry beside it `→ superseded by X`. **★ Rescan the structure tree too (added 2026-07-04)**: incrementally update `<project>-structure` against the real code (the skeleton is a snapshot from the day it was scanned — skip the rescan and it inevitably drifts; scanning directories/class lists is purely mechanical work, scriptable and handable to the overseer). **The arbitration order is fixed: code > structure tree > module map > leaf** — whichever conflicts with its upstream is the one that gets fixed, never the reverse.
4. **Reorder the index** —— MEMORY.md: core rules → ⚑ high-weight (read first) → regular, with `⚑N` weights kept in sync.
5. **Push the backup** —— once tidied, push the memory library to the cloud, and remind/opportunistically push the project library too; otherwise you lose the latest memory when switching machines. **Note: the step of pushing the memory library, if run by me (the agent), gets blocked by the safety classifier as "memory exfiltration," so have the user run it manually in the terminal.**
6. **Verify the scheduled consolidation is still alive (added 2026-07-04)** —— check the in-session scheduled-task list once: the scheduled consolidation hangs off **the very session that created it**, and it silently dies once the original window is closed (the failsafe has no failsafe of its own). If the list comes up empty and the current session isn't the one carrying the task → remind the user to re-hang it.
7. **Clear the workspace and open a new one** —— once tidied, **open a brand-new workspace** (new session); the old window's records can be closed/cleared, because the important info is all in the library, so resuming is lossless.
8. **End with a memory-improvement proposal (added by the user 2026-06-23) + introspection reconciliation (added 2026-07-04)** —— at the **end** of each consolidation: ① check whether the bloodline has any **[introspection] entries left hanging unresolved** (the ones parked by step 3 of the four-step loop in [[growth-memory-refine-on-mistake]]); if so, handle them or state explicitly they're still owed; ② if I have a better idea to improve the memory system itself (structure/rules/layering/weighting/retrieval/dedup methods…), proactively raise it for the user to review (same spirit as [[confirm-plan-before-acting]]'s "proactively propose a better solution," just aimed at the memory system). If there's none, say so — don't force one.

## How to judge similar/duplicate/version-turnover/contradiction (library-tidying criteria)
Extract each memory into **topic (which system/file/concept/rule it's about) + assertion (what is true / how to do it) + scope + time/weight**; what you compare is "topic + assertion." Handles for judging **same topic**: the description points at the same object, the body cites the same file:line / field / id / concept, the `[[links]]` point at each other or share a group, key entities overlap.
- **Similar/duplicate** = same topic + assertions agree or subsume each other → **merge** (union the details, keep the strongest phrasing, **add weights**, take the latest time, delete the old entry).
- **Version turnover** = same topic, the assertions differ because **reality changed** (code changed / decision changed / data updated), the old one was once correct and got superseded by the new → **replace** (new one wins, fold in the still-valid details from the old entry, delete the old). **This is not a contradiction — don't go "correcting" it, just iterate.**
- **Contradiction** = same topic, the two assertions **cannot both be true logically** and both claim to "hold right now" (not one-old-one-new) → **must be eliminated**: ① check the code/repo for current reality; ② if the code can't settle it, go by time (the more recent wins); ③ fix the wrong one, keep the weight. **High weight ≠ correct.**
- **Iteration vs. contradiction distinction**: if it can be explained by an "old → new supersedes" timeline → iteration; if both want to be true right now and are mutually exclusive and can't be reconciled → contradiction (check the code to settle true/false).

## Second-layer memory: four kinds
**① Pitfall archive `pitfalls/`** records pitfalls to avoid them (see below, pulled on demand); **② Dialogue records** keep the user's words + surrounding context verbatim entry by entry (curated, pulled on demand, contradiction removal in consolidation per step 3); **③ Open-issues log** records only OPEN items (added by user 2026-06-20); **④ Full-conversation record `conversations/<date>.md`** —— ⚠️ **superseded by ⑤'s checkpoint direct-write, redundant and can be retired** (user 2026-06-27): the original design was to first dump a full transcript and sort it out afterward; now it's changed to writing directly into each topic log, so this big intermediary file is no longer needed. **⑤ Event bloodline `conversations/events/<event-slug>.md` = a pure chronologically-numbered log (user 2026-06-27; template see [[memory-topic-by-project-structure]] event template · log + index)** —— **checkpoint direct-write**: every time a block is finished / a decision is finalized, write it **on the spot** — the bloodline appends one numbered entry `[N]` (🗣️ user's original words kept in full + 💬 my condensed version), and the corresponding `## feature point` in the leaf updates its `🩸 dialogue ▸ N` index. **No pinned/process split** (classification is left to the leaf index; the log just does pure recording). The big full-conversation file `conversations/<date>.md` is **redundant, can be retired**. Consolidation only does: **fill in what wasn't direct-written + dedup and verify** (don't act as a mover). **Cross-session memory topics** → the dedicated log `conversations/events/记忆模型优化.md` (criterion: changing my memory/habits/behavior = record here, changing code/data = a work event). **Loading = precisely retrieve those few entries by the leaf index** (no matter how old the decision, it won't be lost). → cures "distillation dropped how we talked it through / lost yesterday's conversation" + "reading the latest N entries loses early finalized ones." —— **read first when opening a new window** (hence the hook left at the top of MEMORY.md, not purely on-demand); **consolidation update**: add this window's new opens, delete resolved/abandoned/iterated ones, reconcile and dedup against the first layer's "pending/remaining" (this log = the sole authority on open items); **before finalizing an update, go through the list with the user to confirm "whether it holds"** (still valid / already resolved / changed) before setting it.

## Pitfall archive pitfalls/ (key point: record pitfalls to avoid them, not to replay steps)
**Function**: when a new requirement on a **similar topic** comes up, use this to steer clear of pitfalls stepped in before. One `memory/pitfalls/<date>-<task-slug>.md` per substantive task, **not entered into MEMORY.md** (pulled on demand); from the relevant topic memory's body, hang one line `▸ pitfall: pitfalls/xxx.md`, relying on **topic recall → follow the link to the pitfall**.
**Structure of each pitfall (★ line 4 is the most valuable):**
> what I assumed/did · how it surfaced · true cause · **★ why I got fooled (the misleading signal — next time use it to catch this early)** · fix · (elevate into a distilled memory? [[..]]; on recurrence, weight +1)
**Conventions**: ① date it; ② a pitfall's lesson keeps longer than a command, but still "verify against current code before reuse"; ③ **consolidation only reads/hangs links to pitfalls, never rewrites them** (it's a historical scene — editing it distorts it — treated differently from distilled memory's "check code to correct contradictions"); ④ **a recurring or important pitfall → elevate into one distilled (often high-weight) memory as an "iron rule for avoiding it," weight +1** (the pitfall archive feeds the weighting system).

**The reality of automation (be honest, don't pretend it's fully automatic):**
- "Tidy the current window's contents" must be done by **this very session** — a timer / another session can't read this window's real-time conversation.
- "Clear/reset the workspace" — **no timer can clear the window you're actively using**; "a brand-new workspace" = you open a new session (consolidation has already made that lossless).
- The only part that can run on a pure timer is "memory-library tidying (merge/weight/check-code-and-correct)," and it must be scheduled **locally** (memory is local, a cloud routine can't touch local memory).
- In practice: each day, say "**consolidate**" at the session boundary and it runs per this rule (= a beefed-up version of "flush memory before restart").
