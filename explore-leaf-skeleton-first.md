**挂 [[habit-action-hooks]] 钩子链·主题执行**(动作触发)。

## 铁律(触发 → 动作 → 递归)
- **触发点**:一个需求/叶,在**现有记忆树里找不到它对应的上级树枝**(归属缺失、那块还是黑图)。
- **动作**:**别脑补归类** → 去**查真实代码结构**,找它的上级是谁(类名)。
- **递归往上**:那个上级若在**现有树里也没归属** → **继续往上**(代码里更上一级)→ **一直爬到找到正确的"现有归属"**(连回现有树里某个已存在节点)为止;沿路缺的中间节点立成骨架占位。
- ⚠️**不是"每次都建到根"** —— 是**爬到最近的现有归属就停**(增量嫁接、不重建整条谱系)。树就这样从叶往上、一节节锚在已有节点上长出来。

## 怎么挖(代码=唯一权威)
- **往上找上级**:读类的继承声明(不同语言各自的父类声明语法)拿父类;读**加载器**(谁被 require/注册=活)+ 找操作它的系统类/管理器类。
- **每爬一级问一句**:这级在**现有记忆树里有归属吗?** 有 → 嫁接、停;没有 → 骨架占位、再往上。
- 连回现有树后,沿途新建中间节点的核心逻辑可**占位待建**(先有骨架,别抢着画全图;接 [[premature-abstraction-and-overdistill]] 先具体后抽象)。
- ⚑**别脑补归类 = 瞎猜**:我曾差点把"某个新需求算叶还是枝"纯靠脑补纠结,用户点破:缺的是树枝/树干、层级未知,该去查真实代码结构往上嫁接。

## 找不到怎么办 = 三层排查(便宜的先查,扎在真实代码/文字里,别脑补)
> ⚑根永远是地板的"抽象 base case"是错的脑补(用户 2026-06-27 否);真实情况只有下面三种,**从最便宜往最贵 escalate**。

- **① 错别字?(最便宜·先查)** — 用户常打错字,一个错别字就让词对不上、整条找不到。→ **先轻点确认/纠正**,往往一下就通。接 [[read-user-input-carefully-flag-typos]]。
- **② 走到"烂枝"了?(代码路径错)** — "找不到"往往是顺着一条**没及时剪的死/坏分支(烂枝)**爬,本就接不到。→ **回同模块查有没有其他分支**(别在一条烂枝上吊死);同模块分支都走完还不行 → **标记这条烂枝 + 提醒用户是否修剪**(标"未完成"、**等他决策**,不擅自剪)。
- **③ 真接不到根?(分支全走完仍接不上)** — 那就**不是代码问题、是需求/语义问题**。→ **重新理解需求语义**;还接不上 → **带火气直白点破**:「你这需求有问题、白烧一摞 token」或「我没懂你在说啥,说详细点」——**别软绵绵地客气**(接 [[blunt-frustration-not-polite]]:客气他记不住)。

## 副产品:常逮到矛盾
反向核代码立骨架时,经常顺手撞出**过时注释 / 多套并存 / 张冠李戴**(例:某处注释说"X 继承 A",实际 X 继承 B、A 文件已不存在=化石注释)。→ 配 [[flag-obvious-contradictions]] 当场 flag。
