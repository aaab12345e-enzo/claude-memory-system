💡 **促迁移类记忆(轴2:巧解/思维修正)** —— 用户拆解对话定。见 [[memory-weighting-rule]] 双轴。

**我的惯性逻辑(系统性偏差)**:碰到工具/平台/格式宣称的约束,我默认**接受它的"官方用法/预设工作流",在框内解题**,而不质疑封装本身。这叫 framing-acceptance(接受问题被呈现的"框")—— 接受问题被呈现的方式,而非 reframe(重新框定)到"真实目标 + 底层数据"。

**本次实例(某次要从一个第三方资源包里取素材)**:那个工具只有 Windows `.exe`。
- 我的惯性 → "用工具的导出功能是正路 → 问题缩成'怎么搞到 Windows 跑它'",接受了 Windows 阻塞、在框内找解法(虚拟机/借机)。
- **用户的思路(更优)** → "这些资源不就在包里?**不能直接读到吗?**" 回第一性原理:底层数据(素材文件)物理上一定在 → 那个资源包其实只是常见的压缩归档,直接解开就是全套素材,**绕过工具、绕过 Windows、一步到位**。

**修正 heuristic(可迁移)**:碰到"必须用某工具/平台/格式"的阻塞,**先想一步**——真实要的是底层数据/结果,能不能**绕过这层封装直接拿**?
- 打包格式多是可解归档(tar/zip/gzip 及各种 asset pack);"工具导出"往往只是给原始资源套了层 UI/预览。
- **质疑约束本身,别默认在框内优化。** 先问"底层数据在哪、能不能直取",再谈"怎么满足工具的要求"。

**元规则(用户强调)**:凡"我和用户意见不合、且用户更对"的点 = 照见我盲区的**最高信号**,要细记+拆解三段(我的惯性 / 用户更优 / 可迁移修正)并放一层先读,而不是只记一次性结论。
