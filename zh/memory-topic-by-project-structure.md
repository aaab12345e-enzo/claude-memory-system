用户 2026-06-25 定的**记忆检索结构化规则**,根治我反复犯的「同名概念跨项目张冠李戴」——把两个不同项目里同名的概念弄混(例:同一个源文件名在两套项目里各有一份、指的却是两套完全不同的设计)。

**主题命名 = `项目-模块-[事件]`**(用户 2026-06-25 细化):
1. **项目** — 每个独立代码库/产品各算一个项目。
2. **模块** — 所属子系统(例:某项目里的 `<module-a>` / `<module-b>`…)。**每个模块有一张「地图」记忆**=核心逻辑链 + 跟周围模块的连线(触手);代码基本定死、特殊事件改了核心逻辑才更新。名单见 [[module-maps]]。
3. **事件** — 该模块下具体做的那件事(如 `<module-a>-[某次改动]`)。**事件记忆只记该次的 问题/矛盾/错误/总结/判断 + 链回模块地图**,不重复设计。
4. **二层关联**(由项目派生):各项目 → 各自的遗留问题日志 / `pitfalls/`。

**💡字段层模型 + 防膨胀合并规则(用户 2026-06-26,推自技能管线「钩子=触发器树叶节点」)**:模块树再往下,**叶 = 字段**;字段里 **枚举项字段(例:某种带 type/source 的枚举定义)= 改动最频繁 + 逻辑最复杂的枢纽**,因为它**跨树枝关联**(枚举定义在别枝、增删改一个枚举值会 ripple 到所有消费它的枝)。**合并规则**:落到「字段增删改查」的事件 → **合并进模块地图、不单独成叶**(树不膨胀);只有**枚举字段枢纽**(跨枝、改一处动全身)才值得**单独成叶 + 高权重 + 画触手连线**。→ 修剪/合并树叶事件时:大量字段CRUD事件压缩进地图,只追枚举枢纽。延伸 [[module-maps]] 的「核心逻辑+触手」= 触手主要长在枚举字段上。

**子模块分裂线(用户 2026-07-04 定)**:原设计有「项目-模块-**子模块**」一级,实现上先压扁为 项目-模块→事件(子模块层次靠地图内段落撑)。**维持压扁、按内容量分裂**:一张模块地图下事件叶 **≥5 片**、或地图正文**超 100 行** → 该模块该分裂出子模块层;到线时我提出、用户拍板。树深度跟内容长,不预铺。

## ★事件模版 = 叶(结论+索引) + 血脉(纯时序日志) 一对(用户 2026-06-27 定稿·日志+索引架构)

一个事件 = **两文件配对**,经典「**日志 + 索引**」:日志傻而全、索引在叶。

**📄 叶 `项目-模块-事件.md` = 结论 + 索引**:决策/定稿/根因/判断,**去重、迭代、维护到最新**(=现在该信什么)。**每个 `## 功能点` 标题挂一组对话条号索引** `🩸对话 决策▸N · 过程▸M,…`,指向血脉里产生它的那几条。结尾仍挂 `▸ 血脉: conversations/events/<slug>.md`。

**🩸 血脉 `conversations/events/<slug>.md` = 纯时序对话日志**:按时间**编号 `[1][2]…` 纯记录,不分类**(决策还是过程,判定交给叶的索引;日志只管傻、纯、全)。每条 = 🗣️用户原话 + 💬我的理解/动作。

**🔒 两条铁律**:① **不重复**——同决策,叶存"维护到最新的定论"、血脉存"用户原话+为什么";叶=现行真相、日志=来历,别抄两遍(光看结论画不出蓝图,日志补"当时怎么聊出来的")。② **精简只砍我自己的**——🗣️用户 每句原话**一律全留**,只有 💬我的话能精简(我反复犯"过度提炼吃掉用户原话")。

**✍️ 怎么写 = checkpoint 直写**:每干完一块/定稿一决策,**当场**——血脉追一条编号、叶对应 `##` 更新索引条号。**别**先堆全程大文件再事后分拣;`conversations/<日期>.md` 全程大文件冗余、可退役。

**📥 怎么加载 = 按索引精准取**:命中某 `## 功能点` → 顺它的 `🩸对话▸N` **只加载那几条**(决策条+过程条),不必读整份血脉。**取代"大文件读最新N条"**:靠索引精准取、决策再老索引指着也丢不了(堵 recency≠importance)。

**🌐 跨会话主题(如记忆模型优化)= 专门一份日志**:对话散在多次会话的主题(优化我记忆/习惯)→ 建**一份跨会话记忆日志** `conversations/events/记忆模型优化.md`(checkpoint直写、跨会话累积、编号);其"叶"**分布在多个元记忆**里、各自挂 `🩸对话▸N` 索引指进去,**不靠文件名去各处找**。判据:改**我的记忆/习惯/行为**→记忆日志;改**代码/数据**→工作事件日志。

**匹配协议(对话里某字段命中主题时)**:
1. 按 **项目 + 模块**(+事件)匹配。
2. **唯一命中** → 读该**模块地图**(拿全方位结构)+ 相关事件叶 + 二层;**命中叶里某 `## 功能点` → 顺它的 `🩸对话▸N` 索引,从血脉日志精准加载那几条原话**(用户 2026-06-27 日志+索引),据此答/做。
3. **没全中 / 命中多个项目或模块** → **必须先确认**(别猜哪个项目/模块就动手)。

**Why**:同一概念在两项目是两套设计——例:某个机制(例:某模块的某逻辑)只属于其中一个项目,另一个项目的对应模块里这套完全 N/A。不带「项目」维度匹配就反复弄错(我已多次犯同名概念张冠李戴)。

**文件头分类(用户 2026-06-25·两套合一套)**:**主分类**=① `category`(类别):习惯/执行/成长 ② `node_type`(树位置):根 root / 干 trunk / 枝 map / 叶 event。**旧 `type` 降级** → 改名 `tag`、**只留在叶(事件)上**(值=项目/反馈/参考/用户,**供反向查找**:树体检 / 搜索按标签反查);**根/干/枝不带 tag**。

**How to apply**:① 写**新**记忆按 `项目-模块-[事件]` 命名;模块核心逻辑入**模块地图**、事件只入**问题/判断/坑+链回地图** ② 检索先用 项目+模块 对齐、不止看关键词 ③ 歧义(没全中/多中)立刻确认。延续 [[memory-weighting-rule]]「命中高权重必读全文」+ [[translate-english-in-output]];本条管**项目/模块维度去歧义 + 多中/没中要确认**。⚠️旧记忆**碰到哪个模块就顺手按「地图+事件」拆**,不批量(见 [[module-maps]])。
