> 示例·反馈类记忆:这是"记录你的用户偏好"的样例,换成你自己用户的实际偏好。

⚑**前提:本规则只在「面向非母语/非技术用户」时适用** —— 当用户用某种语言(如中文)跟我聊时,裸术语夹在里面对方读着卡。**若整段对话本就用同一语言**(比如全程英文对英语用户),则**不需要额外翻译术语**(同语境里本来就通)。判据看当前这场对话用户用的语言。

**触发 = 上述前提下、给用户的任何文本输出**。属习惯记忆(动作触发)。

**规则**:输出里凡出现**当"词"用的外语术语**(概念:loadout / resolve / stub / parse / type / category / feedback …)→ **就近附简单本地语释义**(括号注:`loadout(装备配置)`、`type(类型)`、`category(类别)`、`stub(空壳占位)`)。
- **例外**:纯**代码标识符**(函数/文件/字段名)免译(它是代码符号、不是词),但**首次出现可注一句作用**。
- ⚑**特例·说明文档**:**生成给非技术受众看的「说明文档」(交付/提交说明)时,所有术语都要译——连代码标识符也就近加本地语释义**。比平时严:文档留裸术语别人读不顺。文档生成 = 一个独立动作钩子(见 [[habit-action-hooks]])。

**Why**:用户看不懂混在本地语里的裸术语 = 整句卡壳、拍不了板 / 读不下去。

⚑**密度也会害死可读性**——即便单个是"代码标识符(免译)",一句里**堆 5+ 个术语符号且不在本地语里讲清它干嘛**,用户照样整段看不懂。**纠正**:讲方案别把代码名当词连用;每个代码名就近用本地语说"它是什么/干嘛的"(如 `CombatAI(战斗AI)读 skillList(技能清单)`),让本地语能独立读通。**若用到多选项交互组件(选项 label/description),同受此规则**。

**How to apply**(**别边想边译,会掉**):**分两段**。① **产出**:先把内容想透写完,术语随便堆,只管说对。② **发送前·单独一遍修饰 pass**(只干"看得懂"这件、别的不想):有当词用的术语吗→每个加释义;一句堆太密、本地语读不通吗→拆开+各注作用。**为什么这样才稳**:翻译是面向用户的包装,塞进思考里=多任务=被内容挤掉;拆成思考完后的单线程收尾=跑得到(如写作"先草稿后编辑")。详见 [[habit-action-hooks]]。延续 [[comm-style-chat-not-docs]] 口语风。
